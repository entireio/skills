# Investigation: new-skill-prompt

**Status:** confirmed
**Started:** 2026-05-19

## TLDR

Entire misses sessions in this workflow because the failure is two-sided,
not just a skill gap. Producer side: `/Users/alisha/Projects/devenv` is its
own non-Entire-enabled repo, so a parent-launched run has no repo session
state for Entire to attach from. Consumer side: child repo hooks, `session
current/list`, and `session attach` all resolve session state through the
cwd's git common dir, so they cannot discover a session registered somewhere
else. Strongest evidence: live `entire session list --json` and `entire
session current --json` from `devenv/` return no sessions, while the CLI
source (`session/state.go:379-386`, `strategy/session_state.go:78-114`,
`hooks_git_cmd.go:67-75`) shows the reader/hook paths are repo-scoped; a
skill can attach a known session id into child repos, but the exact
zero-prerequisite workflow needs a repo-independent current-session-id
source before the skill can be reliable.

## Question

Context: I use entire to track all agent sessions but entire is repo based
Problem: I often run my agents in a higher level folder so it can work across repos. For example /Users/alisha/Projects/devenv then have agents make changes across different folders/repos opening worktrees. But then my agent sessions don't get tracked by entire in the repos
Solution: is there a skill you can build that fixes this issue? it should use entire commands to do this. if it needs a new entire command let me know and I will consider building it.

## System under investigation

```
Producer side:

  Agent CLI launched from /Users/alisha/Projects/devenv
        |
        v
  Entire session hooks resolve cwd's git repo/common dir
        |
        v
  /Users/alisha/Projects/devenv is not Entire-enabled
  (.entire/settings.json missing; .git/entire-sessions absent)
        |
        v
  No parent session state for `entire session current/list` to find

Consumer side:

  Agent edits/commits under child repos:
      devenv/cli/        devenv/entire.io/        devenv/entiredb/
      (each has its own .git and .entire/settings.json; all are

        |
        v

  git common dir:
      <child>/.git/entire-sessions/*.json
        |
        v
  Parent-launched session is absent from the child store, so no
  Entire-Checkpoint trailer is linked to the child's commits.

Manual recovery primitive:

  known session id + target child repo
        |
        v
  (cd target && entire session attach <session-id> --force)
        |
        v
  Reads agent transcript, redacts/writes checkpoint metadata, amends
  target repo HEAD with Entire-Checkpoint trailer
        |
        v
  Cost amplification on bad discovery/retry:
      per target attempt = transcript read/redaction + checkpoint write/fetch
      + possible commit amend of the wrong repo's HEAD. Existing CLI has no
      --target-repo, --dry-run, or repo-independent "current agent session"
      command to bound that risk.

## Approach

Inspected the Entire CLI surface for `session list`, `session current`,
`session info`, and `session attach`; verified behavior from three cwd
locations (`scope-skill`, `/Users/alisha/Projects/devenv`, and
`/Users/alisha/Projects/devenv/cli`); and read the CLI source for session
state storage, current-session selection, and attach. Also verified the
user's actual directory layout: `/Users/alisha/Projects/devenv` is a git
repo without `.entire/settings.json`, and `cli/`, `entire.io/`, and
`entiredb/` are gitignored independent repos with their own `.git` and
`.entire/settings.json`.

No destructive commands were run. I did not run `entire session attach`
because it can amend the target repo's HEAD.

## Findings

### 1. The `devenv/` parent is a separate, non-Entire-enabled repo; the child repos are gitignored independent repos

Evidence:

- `/Users/alisha/Projects/devenv/.git/` exists, so `devenv` is a real git
  repo.
- `/Users/alisha/Projects/devenv/.gitignore:1-4` lists `cli/`,
  `entiredb/`, and `entire.io/` under "Cloned repositories".
- `/Users/alisha/Projects/devenv/.entire/` contains only `logs/`; no
  `.entire/settings.json` exists.
- `/Users/alisha/Projects/devenv/.git/entire-sessions` does not exist.
- `cd /Users/alisha/Projects/devenv && entire status --json` returned
  `{"enabled":false,"agents":null,"active_sessions":null,"error":"not set up"}`.
- `cd /Users/alisha/Projects/devenv && entire session current --json`
  returned `No active session found in this worktree.`
- `find /Users/alisha/Projects/devenv -maxdepth 3` found child repo
  settings at:
  - `/Users/alisha/Projects/devenv/cli/.entire/settings.json`
  - `/Users/alisha/Projects/devenv/entire.io/.entire/settings.json`
  - `/Users/alisha/Projects/devenv/entiredb/.entire/settings.json`

Conclusion for this finding: checkpoints need to land in each child repo's
commit history. The parent repo's commit history is the wrong target because
the actual code commits happen in gitignored child repos.

### 2. The current CLI does not have a global session registry; `session list` is scoped to the current repo's git common dir

This disputes the first draft's central claim that `entire session list
--json` is global across unrelated repos.

Evidence:

- `cd /Users/alisha/Projects/devenv && entire session list --json` returned
  `[]`.
- `cd /Users/alisha/.marvin/worktrees/skills/scope-skill && entire session
  list --json` returned only skills-repo sessions, including
  `ff4ae70d-2f63-4057-a175-1183b6fe1f8e`.
- `cd /Users/alisha/Projects/devenv/cli && entire session list --json`
  returned cli-repo sessions, including active/idle sessions rooted at
  `/Users/alisha/Projects/devenv/cli` and
  `/Users/alisha/.marvin/worktrees/cli/entire-labs-investigate`.
- Source confirms the behavior: `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/sessions.go:340`
  calls `strategy.ListSessionStates`, and
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/session/state.go:377`
  creates the store from `getGitCommonDir`.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/session/state.go:609`
  runs `git rev-parse --git-common-dir`, then stores sessions under
  `<common-dir>/entire-sessions` at
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/session/state.go:384-385`.
- The apparent cross-worktree visibility inside `devenv/cli` is explained
  by shared git common dir, not by a global registry:
  `git -C /Users/alisha/Projects/devenv/cli rev-parse --git-common-dir`
  returned `.git`, and
  `git -C /Users/alisha/.marvin/worktrees/cli/entire-labs-investigate
  rev-parse --git-common-dir` returned
  `/Users/alisha/Projects/devenv/cli/.git`.

Conclusion for this finding: a skill invoked from `devenv/` cannot recover
the current parent-launched session by asking a child repo's `entire session
list`. The child repo cannot see the parent's store, and the parent currently
has no store entries.

### 3. Producer failure: if the agent starts in the non-enabled parent, there may be no Entire session id to attach

Evidence:

- `/Users/alisha/Projects/devenv/.git/entire-sessions` is absent; a direct
  `find .git/entire-sessions -maxdepth 1 -type f` from the parent failed
  with `No such file or directory`.
- `entire session current --json` from the parent returned no active session,
  and `entire session list --json` from the parent returned `[]`.
- `FindMostRecentSession` is intentionally current-worktree oriented:
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/strategy/session_state.go:91-114`
  scopes by `paths.WorktreeRoot` when possible.
- `paths.WorktreeRoot` is based on `git rev-parse --show-toplevel`
  (`/Users/alisha/Projects/devenv/cli/cmd/entire/cli/paths/paths.go:87-115`),
  so a launch from `devenv/` resolves to the parent repo, not to a child
  repo the agent later edits.

This is the producer side of the failure. The agent session must be created
somewhere before `entire session attach <session-id>` can be used reliably.
If `devenv/` is not Entire-enabled and the agent runtime does not expose its
native session id through an Entire command, the skill has no trustworthy
session id to pass to `attach`.

### 4. Consumer failure: child repo hooks and `attach` read/write only the target repo, so they cannot infer a parent-launched session

Evidence:

- `entire session attach --help` says it "creates a checkpoint from the
  session's transcript and links it to the last commit"; no `--target-repo`,
  `--commit`, or `--dry-run` flag is exposed.
- Hook logging is repo-gated and repo-scoped: `hooks_git_cmd.go:67` returns
  immediately when the cwd repo is not set up/enabled, and
  `hooks_git_cmd.go:75` asks `strategy.FindMostRecentSession(ctx)` for the
  session id in that same cwd.
- `session current` follows the same consumer path:
  `session_current.go:42-44` calls `strategy.FindMostRecentSession(ctx)` and
  prints "No active session found in this worktree" when it returns empty.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:91` exits early
  only if the target repo is disabled; otherwise all work is for the current
  cwd.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:576-594`
  validates the session against the current repo and opens the current repo's
  session store.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:606-617`
  resolves and reads the transcript after a session id is already known.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:489-501`
  reuses a checkpoint trailer already on HEAD or generates a new checkpoint
  id, and `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:730-734`
  amends the current repo's HEAD commit.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:581-582`
  requires at least one commit in the target repo.

Consumer-side cost/risk per bad attempt: once a skill guesses a target repo
and session id, attach can read/redact the transcript, write checkpoint
metadata, fetch checkpoint metadata if HEAD already references a checkpoint,
and amend the target repo's last commit. That makes "try a bunch of child
repos until one works" the wrong failure mode; the skill needs strong target
repo and session-id evidence before invoking attach.

Symmetric consequence (verified): even if the parent `devenv/` were
Entire-enabled with a registered session, child repo git hooks would still
not pick it up automatically. `strategy.FindMostRecentSession`
(`devenv/cli/cmd/entire/cli/strategy/session_state.go:95-114`) and
`strategy.ListSessionStates`
(`devenv/cli/cmd/entire/cli/strategy/session_state.go:78-89`) both construct
the store via `session.NewStateStore`, which resolves the git common dir of
the hook's cwd — i.e., the child repo at commit time. Parent-enabled by
itself solves only the producer half; the consumer (child repo hook /
attach) needs the parent's session id pushed into the child repo's store
before the child's commit hook fires.

### 5. Existing commands can support a constrained skill, but not the exact zero-prerequisite parent-dir workflow

Verified usable pieces:

- If the skill already has a valid session id, it can run
  `entire session attach <session-id> --force` inside each affected child repo.
- If the launch cwd is Entire-enabled, `entire session current --json` can
  expose the current session id and `worktree_path`.
- `entire session info <id> --json` can expose `files_touched` for sessions
  that are in the current repo's session store; confirmed from
  `/Users/alisha/Projects/devenv/entire.io`, where `session current --json`
  included `files_touched` entries such as
  `api/src/lib/transcript-parsers/amp-parser.ts`.

Verified gaps:

- From the user's concrete parent cwd (`/Users/alisha/Projects/devenv`), the
  existing `session current/list` commands return no session data.
- From a child repo, `session current/list` sees only that repo clone's git
  common-dir states, not the parent repo.
- `~/.entire/registered-repos.json` is not a complete discovery source on
  this machine: it contains `/Users/alisha/Projects/devenv/cli`, but not
  `/Users/alisha/Projects/devenv/entire.io` or
  `/Users/alisha/Projects/devenv/entiredb`, even though both child repos have
  `.entire/settings.json` and `entire status --json` reports enabled in
  `entiredb` and `entire.io`.

Conclusion for this finding: a skill can ship only with a clear prerequisite
or explicit input: either the parent launch repo must be Entire-enabled before
the agent run starts, or the user/agent must provide the native session id.
For the exact "I launched from a higher-level folder and now the child repos
did not track it" symptom, existing commands are missing a repo-independent
way to ask "what is the current agent session id?" A `--target-repo` flag
would reduce attach orchestration risk, but it does not solve the missing
session-id producer gap by itself.

### 6. One local CLI/settings mismatch can cause false negatives during repo detection

This is not the root cause of the parent-dir problem, but it matters for a
skill that audits child repos before attaching.

Evidence:

- `/Users/alisha/Projects/devenv/cli/.entire/settings.local.json:1-10`
  sets `"enabled": false` and includes an `"investigate"` object.
- The installed CLI (`Entire CLI 0.6.2`) reports from `devenv/cli`:
  `{"enabled":false,"agents":null,"active_sessions":null,"error":"failed to load settings: loading settings: merging local settings: parsing JSON: json: unknown field \"investigate\""}`.
- The current source explains why: settings use strict JSON decoding with
  `DisallowUnknownFields` in
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/settings/settings.go:469-472`
  and again for local overrides at
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/settings/settings.go:551-557`.
- `checkDisabledGuard` defaults to enabled on settings-load errors
  (`/Users/alisha/Projects/devenv/cli/cmd/entire/cli/setup.go:1199-1204`),
  so `attach` may still run, while `status` reports an error.

Conclusion for this finding: a skill should not use `entire status --json`
alone as the authoritative "can I attach here?" signal. Presence of a git repo
plus `.entire/settings.json`, the actual `attach` preflight result, and a
clear skip/error reason are more reliable than assuming `status` failures mean
"not an Entire repo."

## Unknowns / Assumptions

- **Not reproduced by spawning a fresh agent from `devenv/`**: current
  evidence shows no existing parent session states and source says the
  state store is repo/common-dir scoped. I did not start a new Claude/Codex
  run from `devenv/` to watch the hook lifecycle live.
- **Attach idempotency not exercised**: source shows `promptAmendCommit`
  skips appending an identical checkpoint trailer already on HEAD, and
  attach reuses an existing session state's checkpoint when present. I did
  not run attach twice because it can amend commits.
- **Affected-repo detection remains partly design work**: for a parent-enabled
  future run, `files_touched` may be enough for apply-patch paths under the
  parent, but worktrees opened outside the parent (`~/.marvin/worktrees/...`)
  and shell-created files need separate handling or explicit repo input.
- **CLI command shape is not settled**: the evidence supports a missing
  repo-independent "current agent session id" command as the main gap, plus
  `attach --target-repo` / `--commit` / `--dry-run` as consumer-side
  risk reducers. The exact command names are still a design decision.

## Conclusion

Consensus: the investigation is complete. Existing Entire commands can support
a constrained scope skill only when it already has a trustworthy session id, or
when the parent launch repo is Entire-enabled before the agent run starts and
the skill can explicitly attach that session into the affected child repos. The
commands do not support the exact zero-prerequisite parent-directory workflow
after the fact: the parent has no session state, the child repos cannot see
outside their own git-common-dir stores, and `attach` has no target/dry-run
surface to make blind multi-repo attempts cheap or safe.

- `cd /Users/alisha/.marvin/worktrees/skills/scope-skill && entire session
  list --json` returned only skills-repo sessions, including
  `ff4ae70d-2f63-4057-a175-1183b6fe1f8e`.
- `cd /Users/alisha/Projects/devenv/cli && entire session list --json`
  returned cli-repo sessions, including active/idle sessions rooted at
  `/Users/alisha/Projects/devenv/cli` and
  `/Users/alisha/.marvin/worktrees/cli/entire-labs-investigate`.
- Source confirms the behavior: `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/sessions.go:340`
  calls `strategy.ListSessionStates`, and
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/session/state.go:377`
  creates the store from `getGitCommonDir`.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/session/state.go:609`
  runs `git rev-parse --git-common-dir`, then stores sessions under
  `<common-dir>/entire-sessions` at
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/session/state.go:384-385`.
- The apparent cross-worktree visibility inside `devenv/cli` is explained
  by shared git common dir, not by a global registry:
  `git -C /Users/alisha/Projects/devenv/cli rev-parse --git-common-dir`

  `git -C /Users/alisha/.marvin/worktrees/cli/entire-labs-investigate

Conclusion for this finding: a skill invoked from `devenv/` cannot recover
the current parent-launched session by asking a child repo's `entire session
list`. The child repo cannot see the parent's store, and the parent currently
has no store entries.

### 3. Producer failure: if the agent starts in the non-enabled parent, there may be no Entire session id to attach

Evidence:

- `/Users/alisha/Projects/devenv/.git/entire-sessions` is absent; a direct
  `find .git/entire-sessions -maxdepth 1 -type f` from the parent failed
  with `No such file or directory`.
- `entire session current --json` from the parent returned no active session,
  and `entire session list --json` from the parent returned `[]`.
- `FindMostRecentSession` is intentionally current-worktree oriented:
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/strategy/session_state.go:91-114`
  scopes by `paths.WorktreeRoot` when possible.
- `paths.WorktreeRoot` is based on `git rev-parse --show-toplevel`
  (`/Users/alisha/Projects/devenv/cli/cmd/entire/cli/paths/paths.go:87-115`),
  so a launch from `devenv/` resolves to the parent repo, not to a child
  repo the agent later edits.

This is the producer side of the failure. The agent session must be created
somewhere before `entire session attach <session-id>` can be used reliably.
If `devenv/` is not Entire-enabled and the agent runtime does not expose its
native session id through an Entire command, the skill has no trustworthy
session id to pass to `attach`.

### 4. Consumer failure: child repo hooks and `attach` read/write only the target repo, so they cannot infer a parent-launched session

Evidence:

- `entire session attach --help` says it "creates a checkpoint from the
  session's transcript and links it to the last commit"; no `--target-repo`,
  `--commit`, or `--dry-run` flag is exposed.
- Hook logging is repo-gated and repo-scoped: `hooks_git_cmd.go:67` returns
  immediately when the cwd repo is not set up/enabled, and
  `hooks_git_cmd.go:75` asks `strategy.FindMostRecentSession(ctx)` for the
  session id in that same cwd.
- `session current` follows the same consumer path:
  `session_current.go:42-44` calls `strategy.FindMostRecentSession(ctx)` and
  prints "No active session found in this worktree" when it returns empty.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:91` exits early
  only if the target repo is disabled; otherwise all work is for the current
  cwd.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:576-594`
  validates the session against the current repo and opens the current repo's
  session store.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:606-617`
  resolves and reads the transcript after a session id is already known.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:489-501`
  reuses a checkpoint trailer already on HEAD or generates a new checkpoint
  id, and `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:730-734`
  amends the current repo's HEAD commit.
- `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/attach.go:581-582`
  requires at least one commit in the target repo.

Consumer-side cost/risk per bad attempt: once a skill guesses a target repo
and session id, attach can read/redact the transcript, write checkpoint
metadata, fetch checkpoint metadata if HEAD already references a checkpoint,
and amend the target repo's last commit. That makes "try a bunch of child
repos until one works" the wrong failure mode; the skill needs strong target
repo and session-id evidence before invoking attach.

Symmetric consequence (verified): even if the parent `devenv/` were
Entire-enabled with a registered session, child repo git hooks would still
not pick it up automatically. `strategy.FindMostRecentSession`
(`devenv/cli/cmd/entire/cli/strategy/session_state.go:95-114`) and
`strategy.ListSessionStates`
(`devenv/cli/cmd/entire/cli/strategy/session_state.go:78-89`) both construct
the store via `session.NewStateStore`, which resolves the git common dir of
the hook's cwd — i.e., the child repo at commit time. Parent-enabled by
itself solves only the producer half; the consumer (child repo hook /
attach) needs the parent's session id pushed into the child repo's store
before the child's commit hook fires.

### 5. Existing commands can support a constrained skill, but not the exact zero-prerequisite parent-dir workflow

Verified usable pieces:

- If the skill already has a valid session id, it can run
  `entire session attach <session-id> --force` inside each affected child repo.
- If the launch cwd is Entire-enabled, `entire session current --json` can
  expose the current session id and `worktree_path`.
- `entire session info <id> --json` can expose `files_touched` for sessions
  that are in the current repo's session store; confirmed from
  `/Users/alisha/Projects/devenv/entire.io`, where `session current --json`
  included `files_touched` entries such as
  `api/src/lib/transcript-parsers/amp-parser.ts`.

Verified gaps:

- From the user's concrete parent cwd (`/Users/alisha/Projects/devenv`), the
  existing `session current/list` commands return no session data.
- From a child repo, `session current/list` sees only that repo clone's git
  common-dir states, not the parent repo.
- `~/.entire/registered-repos.json` is not a complete discovery source on
  this machine: it contains `/Users/alisha/Projects/devenv/cli`, but not
  `/Users/alisha/Projects/devenv/entire.io` or
  `/Users/alisha/Projects/devenv/entiredb`, even though both child repos have
  `.entire/settings.json` and `entire status --json` reports enabled in
  `entiredb` and `entire.io`.

Conclusion for this finding: a skill can ship only with a clear prerequisite
or explicit input: either the parent launch repo must be Entire-enabled before
the agent run starts, or the user/agent must provide the native session id.
For the exact "I launched from a higher-level folder and now the child repos
did not track it" symptom, existing commands are missing a repo-independent
way to ask "what is the current agent session id?" A `--target-repo` flag
would reduce attach orchestration risk, but it does not solve the missing
session-id producer gap by itself.

### 6. One local CLI/settings mismatch can cause false negatives during repo detection

This is not the root cause of the parent-dir problem, but it matters for a
skill that audits child repos before attaching.

Evidence:

- `/Users/alisha/Projects/devenv/cli/.entire/settings.local.json:1-10`
  sets `"enabled": false` and includes an `"investigate"` object.
- The installed CLI (`Entire CLI 0.6.2`) reports from `devenv/cli`:
  `{"enabled":false,"agents":null,"active_sessions":null,"error":"failed to load settings: loading settings: merging local settings: parsing JSON: json: unknown field \"investigate\""}`.
- The current source explains why: settings use strict JSON decoding with
  `DisallowUnknownFields` in
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/settings/settings.go:469-472`
  and again for local overrides at
  `/Users/alisha/Projects/devenv/cli/cmd/entire/cli/settings/settings.go:551-557`.
- `checkDisabledGuard` defaults to enabled on settings-load errors
  (`/Users/alisha/Projects/devenv/cli/cmd/entire/cli/setup.go:1199-1204`),
  so `attach` may still run, while `status` reports an error.

Conclusion for this finding: a skill should not use `entire status --json`
alone as the authoritative "can I attach here?" signal. Presence of a git repo
plus `.entire/settings.json`, the actual `attach` preflight result, and a
clear skip/error reason are more reliable than assuming `status` failures mean
"not an Entire repo."

## Unknowns / Assumptions

- **Not reproduced by spawning a fresh agent from `devenv/`**: current
  evidence shows no existing parent session states and source says the
  state store is repo/common-dir scoped. I did not start a new Claude/Codex
  run from `devenv/` to watch the hook lifecycle live.
- **Attach idempotency not exercised**: source shows `promptAmendCommit`
  skips appending an identical checkpoint trailer already on HEAD, and
  attach reuses an existing session state's checkpoint when present. I did
  not run attach twice because it can amend commits.
- **Affected-repo detection remains partly design work**: for a parent-enabled
  future run, `files_touched` may be enough for apply-patch paths under the
  parent, but worktrees opened outside the parent (`~/.marvin/worktrees/...`)
  and shell-created files need separate handling or explicit repo input.
- **CLI command shape is not settled**: the evidence supports a missing
  repo-independent "current agent session id" command as the main gap, plus
  `attach --target-repo` / `--commit` / `--dry-run` as consumer-side
  risk reducers. The exact command names are still a design decision.

## Conclusion

Consensus: the investigation is complete. Existing Entire commands can support
a constrained scope skill only when it already has a trustworthy session id, or
when the parent launch repo is Entire-enabled before the agent run starts and
the skill can explicitly attach that session into the affected child repos. The
commands do not support the exact zero-prerequisite parent-directory workflow
after the fact: the parent has no session state, the child repos cannot see
outside their own git-common-dir stores, and `attach` has no target/dry-run
surface to make blind multi-repo attempts cheap or safe.
